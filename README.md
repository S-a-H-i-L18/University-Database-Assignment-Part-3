# Task 3.2 – Authentication Design

## Task 3.2(a)

### Proposed Two-Factor Authentication (2FA) Scheme

The existing SARS system uses only a username and password, which represents a **single-factor, knowledge-based authentication** mechanism.

To improve security, SARS should implement **Two-Factor Authentication (2FA)** by combining:

* **Factor 1 (Knowledge Factor):**

  * Username and password.

* **Factor 2 (Possession Factor):**

  * A **Time-Based One-Time Password (TOTP)** generated by an authenticator application such as Google Authenticator or Microsoft Authenticator on the user's mobile device.

### How the Scheme Works

1. The student enters their username and password.
2. The server verifies the login credentials.
3. If the credentials are correct, the server requests a six-digit one-time verification code.
4. The student opens the authenticator application on their registered mobile device and enters the current verification code.
5. The server verifies the one-time code before granting access to the Student Portal.

### Why This Improves Security

Using two authentication factors significantly increases the effort required for an attacker to impersonate a legitimate user. Even if an attacker captures the student's username and password, they cannot successfully log in without also possessing the student's registered mobile device that generates the valid one-time password. This additional possession-based factor provides a much stronger level of protection against credential theft and replay attacks.
## Task 3.2(b)

### Applying the Principle of Least Privilege

The Principle of Least Privilege (PoLP) states that every user should be granted only the minimum permissions required to perform their assigned responsibilities.

For the university administrator, the permissions should be configured as follows:

| Permission               | Access                |
| ------------------------ | --------------------- |
| Read student records     | **Granted**           |
| Modify enrollment data   | **Granted**           |
| Access financial records | **Explicitly Denied** |

### Explanation

The administrator needs read-only access to student records to view academic information and write access to enrollment data to manage course registrations. However, financial records are unrelated to the administrator's responsibilities and should therefore be explicitly denied.

### Benefit if the Account is Compromised

If an attacker compromises the administrator's account, the damage is limited because the attacker cannot access confidential financial records. Restricting permissions reduces the attack surface and minimizes the potential impact of unauthorized access.

## Task 3.2(c)

### Cryptographic Primitive for Integrity

The most appropriate cryptographic primitive for verifying that a student's marks record has not been altered is **Hashing**.

### Why Hashing is Appropriate

A cryptographic hash function produces a fixed-length digest that uniquely represents the contents of a data record. If even a single character of the marks record changes, the resulting hash value changes significantly. By comparing the stored hash with a newly computed hash, the system can determine whether the record has been modified.

### Why the Other Options Are Not Suitable

**Symmetric Encryption**

Symmetric encryption protects confidentiality by encrypting data so that only authorized users with the shared secret key can read it. However, encryption alone does not provide a straightforward mechanism for verifying whether stored data has been altered.

**Asymmetric Encryption**

Asymmetric encryption is primarily used for secure communication, key exchange, and digital signatures. Although digital signatures support integrity, simple asymmetric encryption is not the most appropriate primitive when the objective is only to detect accidental or unauthorized modification of stored data.

### Applying Hashing to a Marks Record

1. A student's marks record is created.
2. The system computes a cryptographic hash of the complete record using a secure hash algorithm such as SHA-256.
3. The hash value is stored securely alongside the marks record.
4. Whenever the record is retrieved, the system computes a new SHA-256 hash of the current record.
5. The newly computed hash is compared with the stored hash.
6. If the two hash values are identical, the record has not been altered. If they differ, the record has been modified and its integrity has been compromised.

## Task 3.2(d)

### Non-Repudiation

Non-repudiation is a security property that ensures a person who performs an action cannot later deny having performed that action.

While **integrity** confirms that data has not been modified, it does not identify who created or approved the data. Any user with access to the same hash function can compute an identical hash value. Therefore, hashing alone cannot prove the identity of the person who produced the hash.

### Why Asymmetric Cryptography Is Required

Non-repudiation requires **asymmetric cryptography** because only the legitimate signer possesses the private signing key. The signer creates a digital signature using the private key, and anyone with the corresponding public key can verify the signature. Since the private key is known only to the signer, the signature provides evidence of the signer's identity as well as the integrity of the data.

### SARS Context

**(i) Role Holding the Private Signing Key**

The **University Examination Controller** (or the authorized examination server acting on behalf of the university) should hold the private signing key because this role is responsible for publishing official marks.

**(ii) Role Holding the Public Verification Key**

Students, administrators, and the Student Portal should use the corresponding public key to verify the authenticity of digitally signed marks records.

**(iii) Consequence of Private Key Compromise**

If the private signing key is compromised or leaked, an attacker could generate valid digital signatures for fraudulent marks records. As a result, forged records would appear authentic to anyone verifying them with the public key, completely undermining trust in the authenticity and non-repudiation of examination results.

## Task 3.3(d) – RSA Security

RSA security relies on the computational difficulty of factoring the large integer (n) into its two prime factors (p) and (q). Without these prime factors, an attacker cannot efficiently compute Euler's Totient Function (\phi(n)), which is required to calculate the private key (d).

Although the public key ((e,n)) is publicly available, deriving the private key from it alone is computationally infeasible when very large prime numbers are used. This mathematical difficulty is the foundation of RSA's security.

## Task 3.4(d)

An attacker can observe the public parameters (p), (\alpha), (A), and (B) because they are transmitted over the network. However, the attacker cannot efficiently compute the shared secret key (K) because doing so requires solving the **Discrete Logarithm Problem**, which is computationally infeasible for sufficiently large prime numbers.

When Diffie-Hellman is used without authentication, it is vulnerable to a **Man-in-the-Middle (MITM) Attack**. In this attack, an attacker intercepts the key exchange, establishes separate shared keys with both communicating parties, and transparently relays messages between them without either party realizing that the communication has been compromised.
## Task 3.5 – TLS, Firewalls, and OS Hardening

### Task 3.5(a)

### TLS Handshake

SARS should use **HTTPS** with **TLS** instead of plain HTTP to protect communication between the student's browser and the server.

The TLS handshake consists of the following four main steps:

#### Step 1 – Client Hello

The student's browser sends a **Client Hello** message to the SARS server.

The message includes:

* Supported TLS versions
* Supported cipher suites
* Random client value (Client Random)
* Supported cryptographic extensions

This message informs the server of the client's cryptographic capabilities.

---

#### Step 2 – Server Hello

The SARS server responds with a **Server Hello** message.

The server sends:

* Selected TLS version
* Selected cipher suite
* Server Random value
* Digital certificate containing the server's public key

The browser validates the server's certificate using a trusted Certificate Authority (CA).

---

#### Step 3 – Key Exchange

The client and server perform a secure key exchange (for example, using **Elliptic Curve Diffie-Hellman Ephemeral (ECDHE)**).

Both parties independently compute the same shared secret without transmitting it across the network.

This shared secret is then used to derive the symmetric session keys for encryption and integrity protection.

---

#### Step 4 – Secure Communication Begins

Both the browser and the SARS server exchange encrypted "Finished" messages to verify that the handshake completed successfully.

After successful verification:

* A secure encrypted communication channel is established.
* All subsequent HTTP traffic is transmitted as HTTPS using symmetric encryption.

---

### Result of the TLS Handshake

By the end of the handshake:

* The server has been authenticated.
* A shared symmetric session key has been established.
* Confidentiality, integrity, and secure communication are provided.

## Task 3.5(b)

### Hardware Firewall

The university firewall is configured to allow only:

* Port 80 (HTTP)
* Port 443 (HTTPS)

#### Attack Prevented

The firewall blocks unauthorized attempts to access services running on other ports such as SSH (22), FTP (21), Telnet (23), or database ports.

For example, an attacker attempting to connect directly to an exposed database server on port 3306 would be blocked by the firewall.

---

### Network Intrusion Detection System (NIDS)

Unlike the firewall, a Network Intrusion Detection System (NIDS) examines the content and behavior of permitted network traffic.

For example, the firewall allows HTTPS traffic through port 443. However, if an attacker repeatedly attempts SQL Injection through HTTPS requests, the firewall cannot determine that the requests are malicious because they use an allowed port.

The NIDS analyzes traffic patterns and payload characteristics and can detect suspicious activities such as SQL Injection, port scanning, brute-force login attempts, or abnormal network behavior.

## Task 3.5(c)

### Linux Operating System Hardening

Before exposing the SARS application server to the Internet, the following operating system hardening steps should be applied.

### 1. Install Security Updates

Action:

* Update the operating system using the package manager.
* Install the latest security patches for the Linux kernel and all installed packages.

Benefit:

Reduces known software vulnerabilities that attackers could exploit.

---

### 2. Disable Unnecessary Services

Action:

* Disable unused services such as FTP, Telnet, rsh, rlogin, and other unnecessary network services.

Benefit:

Reduces the attack surface by eliminating unnecessary entry points.

---

### 3. Secure User Accounts

Action:

* Disable or lock unused user accounts.
* Rename or disable unnecessary default administrative accounts where appropriate.
* Enforce strong password policies and Multi-Factor Authentication (MFA) for administrative access.

Benefit:

Reduces the risk of unauthorized access through weak or unused accounts.

---

### 4. Configure the Host Firewall

Action:

* Configure a host-based firewall (for example, UFW or firewalld) to permit only the required services such as HTTPS while blocking unnecessary inbound traffic.

Benefit:

Provides an additional security layer even if traffic reaches the application server.

## Task 3.5(d)

### Differences Between TLS 1.3 and TLS 1.2

### Difference 1 – Faster Handshake

TLS 1.3 reduces the number of round trips required during the handshake compared with TLS 1.2.

**Benefit**

Fewer round trips reduce connection establishment time, improving performance and reducing latency for users accessing the SARS portal.

---

### Difference 2 – Forward Secrecy

TLS 1.3 requires cipher suites that provide **Forward Secrecy**, typically using ephemeral key exchange methods such as ECDHE.

TLS 1.2 allowed cipher suites that did not always guarantee Forward Secrecy.

**Benefit**

Even if the server's long-term private key is compromised in the future, previously recorded encrypted communication cannot be decrypted because each session uses a unique temporary session key.

# Conclusion

Replacing HTTP with HTTPS, implementing TLS 1.3, applying firewall protection, deploying a Network Intrusion Detection System, and performing operating system hardening significantly improve the confidentiality, integrity, availability, and overall security of the SARS application. These measures protect sensitive student information while ensuring secure and reliable access during both normal operation and periods of peak demand.
